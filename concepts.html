ファイルとディレクトリ
コードは「箱の中の箱」に整理されている
核心

プログラムはファイル（書類）とディレクトリ＝フォルダ（箱）で構成されています。Claude Codeに「〜を直して」と頼む時、どのファイルが何をしているかを知ると指示が格段に的確になります。

典型的なウェブプロジェクトの構造
📁 my-project/
📄 index.html ← ユーザーが最初に見るページ
📁 css/
📄 style.css ← 見た目の定義
📁 js/
📄 app.js ← 動作の定義
📁 api/ ← サーバー側の処理
📄 data.js
📄 package.json ← 「材料リスト」設定ファイル
📁 node_modules/ ← 自動生成（触らない）
パス（住所）の読み方
// これはファイルの「住所」= パス /Users/yamada/my-project/js/app.js ↑ ↑ ↑ プロジェクト フォルダ ファイル // 相対パス（今いる場所からの距離） ./js/app.js ← 今と同じ場所のjsフォルダの中 ../images/a.png ← 一つ上のフォルダのimages
Claude Codeへの応用：「index.htmlのボタンを直して」より「index.htmlの84行目にあるsubmitボタンのクリック処理を直して」のほうが圧倒的に速く・正確に直せます。

クライアントとサーバー
インターネットは「注文」と「配達」で成り立っている
核心

クライアント（あなたのブラウザ）が「これを見せて」とリクエストし、サーバー（どこかのコンピュータ）が応答します。このやりとりがWebのすべての基本です。

URLを打ってページが表示されるまで
🖥️ ブラウザ
(あなた)
→
リクエスト
「このページ見せて」
🌐 インターネット
→
到達
⚙️ サーバー
(どこかのPC)
↓  レスポンス（HTML・CSS・JSファイルを返す）
🖥️ ブラウザが受け取り
画面に表示する
重要な概念：HTTPステータスコード
200 OK ← 成功！普通に返ってきた 301 Redirect ← 移転（別URLに案内） 404 Not Found ← そのページは存在しない 403 Forbidden ← 権限がない（見られない） 500 Server Error ← サーバー側でエラーが起きた
Claude Codeへの応用：「エラーが出た」ではなく「コンソールに404エラーが出た」と伝えるだけで、問題の場所が大幅に絞り込まれます。
Q. ブラウザで「500 Internal Server Error」が表示されました。問題はどこにありますか？

フロントエンドとバックエンド
「見える部分」と「見えない部分」の分業
核心

Webアプリはフロントエンド（ブラウザで動く、目に見える部分）とバックエンド（サーバーで動く、データを扱う部分）に分かれています。どちらの問題かを区別できると、Claude Codeへの指示が的確になります。

フロントエンド（見える側）
HTML・CSS・JavaScript
ブラウザの中で動く
ボタン・レイアウト・アニメーション
ユーザーの操作を受け取る
DevToolsで中身を見られる
バックエンド（見えない側）
Python・Node.js・Goなど
サーバーのPC上で動く
データの保存・認証・計算
データベースと会話する
ソースコードは見られない
ログイン処理の例
【フロント】入力フォームを表示・バリデーション ユーザーが「test@mail.com」「pass123」を入力 ↓ 【通信】バックエンドに送信 POST /api/login { email, password } ↓ 【バック】パスワードを検証・セッションを発行 DBで照合 → 一致したら token を返す ↓ 【フロント】tokenを受け取りページ遷移
フロント 「ボタンの色がおかしい」「アニメーションが動かない」

バック 「ログインできない」「データが保存されない」「遅い」

どちらも 「フォームを送信したが何も起きない」
Claude Codeへの応用：「ログインボタンを押しても何も起きない」と言うより「フロントのsubmit関数は動いているがAPIへのPOSTが返ってこない」と言うほうが、解決速度が3倍は違います。

APIとは何か
プログラム同士が会話する「窓口」
核心

API（Application Programming Interface）とは、異なるプログラムが決まった形式でデータをやりとりするための「窓口」です。飲食店の注文カウンターに例えると、メニューから選んで（リクエスト）、料理が来る（レスポンス）仕組みです。

天気アプリのAPI通信の流れ
📱 あなたのアプリ
→
「東京の天気を教えて」
GET /weather?city=Tokyo
🌦️ 気象API
↓  JSON形式で返ってくる
{ "city": "Tokyo", "temp": 24, "weather": "晴れ" }
HTTPメソッドの意味
GET /users/123 ← 取得（読む） POST /users ← 作成（新しく送る） PUT /users/123 ← 更新（全部書き換え） PATCH /users/123 ← 更新（一部だけ） DELETE /users/123 ← 削除
APIキーとは？

APIを使う「合言葉」です。「誰が何回使ったか」を管理するために使います。絶対に公開しない（GitHubに上げない）のが鉄則です。Claude Codeが「.envファイルに書いて」と言うのはこのためです。

Claude Codeへの応用：エラーに「401 Unauthorized」が含まれていたらAPIキーの問題。「429 Too Many Requests」なら使いすぎです。これを知っているだけでClaudeへの説明が変わります。

エラーメッセージを読む
エラーは「問題の場所と種類を教えてくれる地図」
核心

エラーメッセージを怖がる必要はありません。エラーは「どこで」「何が」「なぜ」壊れているかを教えてくれる最も正確な情報源です。Claude Codeにそのままコピペするだけで解決の8割は終わります。

エラーメッセージの解剖
TypeError: Cannot read properties of undefined (reading 'map')
at App.js:42:18
エラー種類 TypeError = 「型」が間違っている。undefinedなのにmapしようとした
メッセージ undefinedの.mapを呼んだ = データがまだ来ていないのに使おうとした
場所 App.js の 42行目、18文字目 → ここを直せばいい
よく出るエラー早見表
SyntaxError ← 文法ミス。カッコや引用符の閉じ忘れ ReferenceError ← 存在しない変数を使った TypeError ← 型が違う（nullにメソッドを呼んだ等） NetworkError ← 通信失敗。URLかネットの問題 CORS Error ← サーバーが別ドメインからのアクセスを拒否 Module not found ← ファイルパスかパッケージ名が間違い
Claude Codeへの応用：エラー文を「なんかエラーが出ました」ではなくそのままコピペしてください。それだけで解決策が格段に変わります。ブラウザのDevTools（F12）→ Consoleタブで確認できます。

ターミナルの基礎
Claude Codeが一番活躍する「黒い画面」への慣れ方
核心

ターミナル（コマンドライン）はGUIのないコンピュータへのテキスト指示インターフェースです。Claude Codeはここで動きます。仕組みを知ると出力の読み方がわかり、「何が起きているか」を把握できます。

よく使うコマンドの意味
ls ← 今いるフォルダの中身を表示 cd フォルダ名 ← そのフォルダに移動 cd .. ← 一つ上のフォルダに戻る pwd ← 今どこにいるか表示 npm install ← package.jsonの材料を全部インストール npm run dev ← 開発サーバーを起動 npm run build ← 本番用に最適化したファイルを生成 git status ← 何が変更されたか確認 git add . ← 全変更をステージング（保存準備） git commit -m "説明" ← 保存
プロセスとは

コマンドを実行するとプロセス（動いているプログラムの単位）が生まれます。npm run devは終わらない（サーバーがずっと動く）プロセスです。Ctrl+Cで止めます。止めずに別コマンドを実行したい時は新しいターミナルタブを開きます。

環境変数（.envファイル）

プログラムに渡す秘密の設定値です。APIキーやデータベースのパスワードを入れます。.envファイルはGitに含めません（.gitignoreで除外）。

# .envファイルの中身の例 SUPABASE_KEY=eyJhbGciOiJIUzI1NiIsInR5c... API_SECRET=sk-proj-xxxxxxxxxxxx DATABASE_URL=postgresql://user:pass@host/db
Claude Codeへの応用：「npm installしたらエラーが出た」と言う場合、エラーの最後の数行が最重要です。「WARN」は無視していいことが多く、「ERR!」から始まる行が本当の問題です。

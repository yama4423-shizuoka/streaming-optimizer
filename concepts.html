ファイルとディレクトリ
コードは「箱の中の箱」に整理されている
核心

プログラムはファイル（書類）とディレクトリ＝フォルダ（箱）で構成されています。Claude Codeに「〜を直して」と頼む時、どのファイルが何をしているかを知ると指示が格段に的確になります。

典型的なウェブプロジェクトの構造
📁 my-project/
📄 index.html ← ユーザーが最初に見るページ
📁 css/
📄 style.css ← 見た目の定義
📁 js/
📄 app.js ← 動作の定義
📁 api/ ← サーバー側の処理
📄 data.js
📄 package.json ← 「材料リスト」設定ファイル
📁 node_modules/ ← 自動生成（触らない）
パス（住所）の読み方
// ファイルの「住所」= パス /Users/yamada/my-project/js/app.js // 相対パス（今いる場所からの距離） ./js/app.js ← 今と同じ場所のjsフォルダの中 ../images/a.png ← 一つ上のフォルダのimages
Claude Codeへの応用：「ボタンを直して」より「index.htmlの84行目のsubmitボタンを直して」のほうが圧倒的に速く正確です。
Q. 「./css/style.css」というパスは何を意味しますか？

クライアントとサーバー
インターネットは「注文」と「配達」で成り立っている
核心

クライアント（あなたのブラウザ）が「これを見せて」とリクエストし、サーバー（どこかのコンピュータ）が応答します。このやりとりがWebのすべての基本です。

URLを打ってページが表示されるまで
🖥️ ブラウザ
(あなた)
→
リクエスト
🌐 インターネット
→
到達
⚙️ サーバー
(どこかのPC)
↓ レスポンス（HTML・CSS・JSを返す）
🖥️ ブラウザが受け取り表示
HTTPステータスコード早見表
200 OK ← 成功 301 Redirect ← 別URLに移転 404 Not Found ← そのページは存在しない 403 Forbidden ← 権限がない 500 Server Error ← サーバー側の問題
Claude Codeへの応用：「エラーが出た」ではなく「コンソールに404エラーが出た」と伝えるだけで問題箇所が大幅に絞れます。
Q. 「500 Internal Server Error」が表示されました。問題はどこにありますか？
3
フロントエンドとバックエンド
「見える部分」と「見えない部分」の分業
核心

Webアプリはフロントエンド（ブラウザで動く・目に見える）とバックエンド（サーバーで動く・データを扱う）に分かれています。どちらの問題かを区別できると、Claudeへの指示が的確になります。

フロントエンド（見える側）
HTML・CSS・JavaScript
ブラウザの中で動く
ボタン・レイアウト・アニメーション
DevToolsで中身を見られる
バックエンド（見えない側）
Python・Node.js・Goなど
サーバーのPC上で動く
データの保存・認証・計算
ソースコードは外から見えない
フロント 「ボタンの色がおかしい」「アニメーションが動かない」
バック 「ログインできない」「データが保存されない」
どちらも 「フォームを送信したが何も起きない」
Claude Codeへの応用：「ログインできない」より「フロントのsubmit関数は動いているがAPIへのPOSTが返ってこない」と言うほうが解決速度が3倍変わります。
Q. 「ページのタイトルの文字を大きくしたい」はどちらの作業ですか？
4
APIとは何か
プログラム同士が会話する「窓口」
核心

API（Application Programming Interface）とは、異なるプログラムが決まった形式でデータをやりとりする「窓口」です。飲食店のカウンターのように「注文（リクエスト）→料理が来る（レスポンス）」の仕組みです。

天気アプリのAPI通信
📱 あなたのアプリ
→
GET /weather?city=Tokyo
🌦️ 気象API
↓ JSONで返ってくる
{ "city": "Tokyo", "temp": 24, "weather": "晴れ" }
HTTPメソッドの意味
GET /users/123 ← 取得（読む） POST /users ← 作成（新しく送る） PUT /users/123 ← 更新（全部書き換え） DELETE /users/123 ← 削除
APIキーとは？

APIを使う「合言葉」です。絶対に公開しない（GitHubに上げない）のが鉄則。Claude Codeが「.envファイルに書いて」と言うのはこのためです。

Claude Codeへの応用：「401 Unauthorized」はAPIキーの問題。「429 Too Many Requests」は使いすぎ。エラーコードを伝えるだけで解決策が変わります。
Q. APIのGETリクエストは何をするためのものですか？

エラーメッセージを読む
エラーは「問題の場所と種類を教えてくれる地図」
核心

エラーメッセージを怖がる必要はありません。「どこで」「何が」「なぜ」壊れているかを教えてくれる最も正確な情報源です。Claude Codeにそのままコピペするだけで解決の8割は終わります。

エラーメッセージの解剖
TypeError: Cannot read properties of undefined (reading 'map')
at App.js:42:18
種類TypeError = 「型」の間違い。undefinedにmapを呼んだ
内容データがまだ来ていないのに使おうとした
場所App.js の 42行目18文字目 → ここを直せばいい
よく出るエラー早見表
SyntaxError ← 文法ミス。カッコや引用符の閉じ忘れ ReferenceError ← 存在しない変数を使った TypeError ← 型が違う（nullにメソッドを呼んだ等） NetworkError ← 通信失敗。URLかネットの問題 CORS Error ← 別ドメインからのアクセスをサーバーが拒否 Module not found ← ファイルパスかパッケージ名が間違い
Claude Codeへの応用：ブラウザのDevTools（F12）→ Consoleタブを開いて、エラーをそのままコピペしてください。それだけで解決策が変わります。

ターミナルの基礎
Claude Codeが一番活躍する「黒い画面」への慣れ方
核心

ターミナルはGUIのないコンピュータへのテキスト指示インターフェースです。Claude Codeはここで動きます。仕組みを知ると出力の読み方がわかり、「何が起きているか」を把握できます。

よく使うコマンドの意味
ls ← 今いるフォルダの中身を表示 cd フォルダ名 ← そのフォルダに移動 cd .. ← 一つ上のフォルダに戻る pwd ← 今どこにいるか表示 npm install ← 依存パッケージを全部インストール npm run dev ← 開発サーバーを起動 npm run build ← 本番用ファイルを生成 git status ← 何が変更されたか確認 git add . ← 全変更をステージング git commit -m "説明" ← 保存
環境変数（.envファイル）

APIキーやDBパスワードなど秘密の設定値を渡すファイルです。.gitignoreでGitから除外します。

SUPABASE_KEY=eyJhbGciOiJIUzI1NiIsInR5c... DATABASE_URL=postgresql://user:pass@host/db
Claude Codeへの応用：npm installのエラーは「WARN」は無視していい。「ERR!」から始まる行が本当の問題です。その行だけをClaudeに送れば十分です。

データベースの基礎
データを「表」に整理して保存する仕組み
核心

データベース（DB）はデータを整理して保存する仕組みです。Excelのようなテーブル（表）に、行（レコード）と列（カラム）でデータを管理します。このプロジェクトで使っているSupabaseもデータベースです。

テーブルのイメージ（usersテーブル）
| id | email | name | created_at | |----|-------------------|--------|---------------------| | 1 | taro@example.com | 太郎 | 2026-01-15 10:00:00 | | 2 | hana@example.com | 花子 | 2026-01-16 12:30:00 | ↑列（カラム） ↑行（レコード）
SQL — DBへの命令文
-- データを取得 SELECT * FROM users WHERE id = 1 -- データを追加 INSERT INTO users (email, name) VALUES ('taro@example.com', '太郎') -- データを更新 UPDATE users SET name = '次郎' WHERE id = 1 -- データを削除 DELETE FROM users WHERE id = 1
Supabaseとの関係

SupabaseはPostgreSQLというデータベースをAPI経由で使えるようにしたサービスです。コードから supabase.from('users').select() と書くと、裏で SELECT * FROM users が実行されています。

Claude Codeへの応用：「データが取得できない」と言う場合、「usersテーブルのselectが空配列を返す」と言うほうが格段に的確です。テーブル名とカラム名を伝えましょう。
Q. データベースで「レコード」とは何ですか？
8
非同期処理（async/await）
「待ち時間」のあるプログラムの書き方
核心

APIやDBへの通信には時間がかかります。その間プログラムが止まらないように「後で結果が来たら続きをやる」という仕組みが非同期処理です。async/awaitはそれを読みやすく書く構文です。

なぜ非同期が必要か
// 同期（悪い例）— APIが返るまで画面が固まる const data = fetch('/api/users') // ← これは待てない console.log(data) // ← まだ来ていないのに使う → undefined // 非同期（正しい）— awaitで「来るまで待って」と指示 async function getUsers() { const response = await fetch('/api/users') ← 待つ const data = await response.json() ← また待つ console.log(data) ← データが来てから実行される }
よくある失敗パターン
// awaitを忘れた → dataがPromiseオブジェクトになる const data = fetch('/api/users') console.log(data) // → Promise { <pending> } と表示される // async関数の外でawaitを使った → SyntaxError const data = await fetch('/api') // ← async関数の中でしか使えない
Claude Codeへの応用：「データがundefinedになる」「Promise {''}と表示される」というエラーは、ほぼ100% awaitの書き忘れです。Claudeに「awaitの問題ですか？」と聞くだけで解決することが多いです。
Q. コンソールに「Promise {''}」と表示された場合、原因として最も可能性が高いのは？

ビルドとバンドル
「開発用コード」を「本番用コード」に変換する仕組み
核心

ビルドとは、開発用のコードを本番環境向けに変換する処理です。バンドルとは、100個のJSファイルをブラウザが読みやすい1〜数個にまとめること。npm run buildがこれをやっています。

ビルドで何が起きているか
【開発時のコード】読みやすさ優先 src/ components/Button.jsx ← JSXを使ったコンポーネント utils/format.js ← ヘルパー関数 pages/index.jsx ← メインページ npm run build ↓ (変換・圧縮・最適化) 【ビルド後】速度・サイズ優先 dist/ index.html ← 1ファイルに統合 assets/main.a3f9b2.js ← 圧縮・難読化済み assets/style.8c2d1e.css
開発環境（dev）
コードが読みやすい
エラーの場所がわかる
変更が即座に反映される
ファイルサイズは大きい
本番環境（build）
コードが圧縮・難読化される
高速・軽量
変更には再ビルドが必要
Vercelにはこちらをデプロイ
Claude Codeへの応用：「ローカルでは動くのに本番で動かない」はビルドの問題であることが多いです。その一言を添えるだけで調査方向が変わります。

認証の仕組み（Cookie・JWT）
「ログインしたことを覚える」仕組み
核心

HTTPは毎回リセットされる仕組みです。なので「このユーザーはログイン済み」を覚えるための仕組みが別途必要です。それがCookieやJWTです。ログイン関連のバグの大半はここから来ます。

認証（Authentication）
「あなたは誰ですか？」
ログイン・パスワード確認
本人確認のプロセス
認可（Authorization）
「あなたは何ができますか？」
管理者か一般ユーザーか
アクセス権限の管理
JWTによるログインの流れ
① ログイン POST /api/login { email, password } ↓ サーバーがパスワードを照合 ↓ OKなら JWT（トークン）を発行して返す ② JWT = 3つの部分を「.」でつないだ文字列 eyJhbGci....eyJ1c2VyaWQ....SflKxwRJSMeK... ↑ヘッダー ↑ペイロード(中身) ↑署名(改ざん防止) ③ 以降のリクエストにJWTを添付 GET /api/profile Authorization: Bearer eyJhbGci... ↓ サーバーが署名を検証してOKなら返す
Cookieとは

ブラウザに保存される小さなデータです。サーバーが「このCookieを持ってきたらログイン済みとして扱う」という使い方をします。JWTをCookieに保存するパターンが一般的です。

Claude Codeへの応用：「ログインしても画面が変わらない」→「JWTが正しく保存されているかLocalStorageを確認して」と一言添えると調査が早くなります。DevTools → Applicationタブで確認できます。
Q. 「認証（Authentication）」と「認可（Authorization）」の違いは何ですか？
11
DNS・ドメインの仕組み
URLがどうやってサーバーにたどり着くか
核心

ドメイン（例: example.com）は人間が読めるサーバーの住所です。コンピュータはIPアドレス（数字）しか理解できないので、DNS（Domain Name System）が「example.com → 192.0.2.1」と変換します。電話帳のような仕組みです。

URLを打ってからサーバーに届くまで
ブラウザに「example.com」を入力 ↓ ① DNSに問い合わせ 「example.comのIPアドレスを教えて」 ↓ ② DNSが答える 「192.0.2.1 だよ」 ↓ ③ そのIPアドレスにリクエスト GET / HTTP/1.1 → 192.0.2.1 ↓ サーバーがHTMLを返す
DNS伝播とは（今日詰まった場所）

DNSの設定変更は世界中のDNSサーバーに徐々に広がります（数分〜最大48時間）。Vercelにデプロイしても「見られない」ことがあるのはこのためです。「DNSが伝播中」という状態です。

よく見るドメイン関連の設定
Aレコード ドメイン → IPアドレス に変換 CNAMEレコード ドメイン → 別ドメイン に変換（Vercelでよく使う） TTL キャッシュの有効期限（秒）。短いと早く伝播する
Claude Codeへの応用：「デプロイしたのにURLが変わらない」「カスタムドメインが反映されない」は設定ミスかDNS伝播待ちのどちらかです。その区別をClaudeに伝えましょう。

12
キャッシュ
「直したのに変わらない」の原因の大半
核心

キャッシュとは、一度取得したデータを一時保存して「次回は保存済みを使う」仕組みです。表示が速くなる反面、更新してもキャッシュが残って古い状態が表示されることがあります。これが「直したのに変わらない」の原因の大半です。

キャッシュの種類と場所
① ブラウザキャッシュ あなたのPCに保存 → Ctrl+Shift+R で強制再読み込みで解決 ② CDNキャッシュ（Vercel等） 世界中のサーバーに保存 → 再デプロイするかキャッシュをパージ ③ データベースのキャッシュ クエリ結果を保存 → TTLが切れるか手動でクリア ④ アプリ内キャッシュ（React等） 状態管理の変数 → ページ再読み込みかstateのリセット
「直したのに変わらない」の診断フロー
Ctrl+Shift+R（強制再読み込み）で直った → ブラウザキャッシュが原因 再デプロイしたら直った → CDNキャッシュが原因 シークレットモードで開いたら直った → 拡張機能かブラウザキャッシュが原因 何をやっても変わらない → コード自体が変更されていない（保存し忘れ等）
Claude Codeへの応用：「修正したのに画面が変わらない」と言うとき、まずCtrl+Shift+Rを試してから報告しましょう。それだけで「キャッシュの問題でした」という無駄なやりとりが消えます。

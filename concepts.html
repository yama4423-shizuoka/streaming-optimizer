バイブコーディング入門

Claude Code の動きを
直感的に追う

コードを自分で書く必要はない。AIが今何をしているかを理解して、正しい方向に導く力がバイブコーディングの本質。その力に必要な14の知識。

バイブコーディングとは
「書く人」ではなく「理解して導く人」になる
必要ないこと
✗コードをゼロから書けること
✗すべての文法を暗記すること
✗アルゴリズムを設計できること
✗エラーを自分で直せること
本当に必要なこと
✓Claude が今何をしているか追えること
✓エラーの意味を読んで正確に伝えられること
✓「どちらの問題か」を切り分けられること
✓次に何をすべきかを判断できること
このサイトの14の知識は、すべて「Claude Code がこれをやっているとき、あなたは何を理解していればいいか」という問いに答えるために選ばれています。各セクションの冒頭には実際のシナリオを載せています。
用語集 — 91語
ファイルとパス
ファイル
テキストやコードを保存する1つの単位。名前と中身を持つ。
例: index.html / style.css / app.js
ディレクトリ（フォルダ）
ファイルをまとめる入れ物。フォルダと同じ意味。階層構造を作れる。
例: src/ / components/ / api/
パス
ファイルの「住所」。どこにあるかを示す文字列。
例: /Users/yamada/project/index.html
相対パス
今いる場所を起点とした住所。「./」は今の場所、「../」は一つ上の階層。
./css/style.css / ../images/logo.png
絶対パス
一番上の階層（ルート）から始まる完全な住所。どこから見ても同じ。
/Users/yamada/project/src/app.js
拡張子
ファイル名末尾の「.○○」部分。ファイルの種類を示す。
.html=ページ .css=デザイン .js=動作 .json=データ
HTML / CSS / JavaScript
HTML
ウェブページの「骨格」を作るマークアップ言語。見出し・本文・ボタンなどの構造を定義する。
<h1>タイトル</h1> / <button>送信</button>
CSS
ウェブページの「見た目」を定義するスタイルシート言語。色・サイズ・配置を指定する。
color: red; / font-size: 16px; / display: flex;
JavaScript（JS）
ウェブページに「動き」をつけるプログラミング言語。ボタンのクリック処理やAPI通信など。
document.querySelector() / fetch() / addEventListener()
フロントエンド
ブラウザで動くUI側の部分。HTML・CSS・JSで作る。ユーザーが直接見て操作する場所。
ボタン・フォーム・アニメーションはフロントエンド
バックエンド
サーバーで動くデータ処理側の部分。Python・Node.js・Goなどで作る。外からソースコードは見えない。
ログイン処理・DB保存・メール送信はバックエンド
コンポーネント
UIの部品。再利用できる画面のかたまり。Reactでは関数で定義する。
Button.jsx / Header.jsx / UserCard.jsx
JSX
JavaScriptの中にHTMLのような記法を書けるReact用の構文。ビルド時にJSに変換される。
return <div className="card">{name}</div>
フレームワーク
アプリ開発の「骨格」となるツールセット。1から作らなくていいよう共通機能をまとめたもの。
Next.js / React / Vue / SvelteKit
DevTools（開発者ツール）
ブラウザに内蔵された開発用ツール。F12で開ける。Console・Network・Elementsタブが特に重要。
F12 → Console でエラー確認 / Network で通信確認
コンソール（Console）
DevToolsのタブの一つ。console.log() の出力やエラーメッセージが表示される。
console.log("確認") → コンソールに表示される
Web 通信・API
ブラウザ
ウェブページを表示するアプリ。Chrome・Safari・Firefoxなど。フロントエンドコードはここで動く。
Chrome / Safari / Firefox / Edge
URL
ウェブページやAPIの「住所」。プロトコル・ドメイン・パスで構成される。
https://example.com/api/users?id=1
HTTP
ブラウザとサーバーが通信するためのルール（プロトコル）。リクエストとレスポンスで成り立つ。
http:// または https:// で始まるURLが使っている
リクエスト
クライアント（ブラウザ）からサーバーへの「お願い」。「このデータを見せて」「これを保存して」など。
GET /users → 「ユーザー一覧をください」
レスポンス
リクエストに対するサーバーからの「返答」。データとステータスコードが返ってくる。
200 OK + JSONデータ / 404 Not Found
ステータスコード
通信結果を示す3桁の数字。200=成功・404=見つからない・500=サーバーエラーなど。
200 OK / 401 Unauthorized / 404 Not Found / 500 Error
API
異なるプログラムがデータをやり取りするための「窓口」。決まった形式でリクエストを送ると結果が返ってくる。
GET /api/weather?city=Tokyo → 天気データが返る
APIキー
APIを使う際の「合言葉」となる秘密の文字列。絶対に公開してはいけない。.envに書く。
OPENAI_API_KEY=sk-xxxxx → .envに書いてGitから除外
エンドポイント
APIの「窓口URL」。特定の機能にアクセスするためのパス。
/api/users (ユーザー一覧) / /api/login (ログイン)
GET / POST / PUT / DELETE
HTTPメソッド。GETは取得、POSTは作成、PUTは更新、DELETEは削除を意味する。
GET /users=一覧取得 / POST /users=新規作成
JSON
データを表す軽量なテキスト形式。APIのレスポンスや設定ファイルで多用される。
{"name": "太郎", "age": 20, "active": true}
CORS
異なるドメイン間の通信をブラウザが制限する仕組み。サーバー側で許可設定が必要。
localhost:3000 → api.example.com にアクセスするとCORSエラー
HTTPS / SSL
通信内容を暗号化する仕組み。本番環境では必須。Vercelは自動で対応してくれる。
URLが https:// で始まるサイトで使われている
localhost
自分のPC上で動くサーバーの住所。インターネットには公開されていない。
http://localhost:3000 ← 自分だけが見える
ポート
通信の「窓口番号」。1台のPCで複数のサーバーを区別するために使う。
localhost:3000 の「3000」がポート番号
ターミナルと npm
ターミナル
テキストでコンピュータを操作する黒い画面。Claude Code はここで動く。
Mac: Terminal.app / VS Code: 下部パネル
コマンド
ターミナルに入力する命令文。Enter で実行する。
ls / cd フォルダ名 / npm run dev
npm
Node.jsのパッケージ管理ツール。外部ライブラリのインストールやスクリプト実行に使う。
npm install / npm run dev / npm run build
Node.js
JavaScriptをブラウザ外（サーバーやターミナル）で動かす実行環境。npm を使うために必要。
node app.js → サーバー側でJSが動く
パッケージ（ライブラリ）
npmで追加できる便利な機能の束。他の人が作ったコードを再利用できる。
react / axios / tailwindcss / supabase
node_modules/
npmがインストールしたパッケージが入るフォルダ。中身は触らない。Gitにも含めない。
.gitignoreに書いて除外するのが鉄則
package.json
プロジェクトの「材料リスト」。使うパッケージとスクリプトが書いてある設定ファイル。
"dependencies" にパッケージ名とバージョンが並ぶ
環境変数（.env）
APIキーやパスワードなどの秘密の設定値。.envファイルに書いてGitからは除外する。
SUPABASE_KEY=xxxxx を .gitignoreで除外
データベース
データベース（DB）
データを整理して保存する仕組み。Excelのような表（テーブル）でデータを管理する。
Supabase / PostgreSQL / MySQL / SQLite
SQL
データベースへの命令文。SELECT（取得）・INSERT（追加）・UPDATE（更新）・DELETE（削除）が基本。
SELECT * FROM users WHERE id = 1
テーブル
データを行と列で管理する表。Excelのシートのようなもの。DBの基本単位。
usersテーブル / postsテーブル / ordersテーブル
レコード（行）
テーブルの1行。1件のデータのまとまり。
id=1, name=太郎, email=taro@... ← 1レコード
カラム（列）
テーブルの列。データの項目名。型（テキスト・数値・日時など）を持つ。
id / name / email / created_at ← カラム名
Supabase
PostgreSQLをAPIやダッシュボードで簡単に使えるクラウドサービス。認証・ストレージ機能もある。
supabase.from('users').select() → SQLを書かずにDB操作
非同期処理
非同期処理
待ち時間中（APIやDB通信中）も他の処理を続ける仕組み。async/awaitで書く。
API呼び出し中も画面が固まらない
同期処理
1つが終わるまで次の処理が待つ方式。通信処理に使うと画面が固まる。
上から順番に1行ずつ実行する通常のコード
Promise
非同期処理の「将来の結果」を表すオブジェクト。awaitで中身を取り出す。
Promise { <pending> } ← awaitなしで使った時のエラー表示
async / await
非同期処理を同期処理のように読みやすく書く構文。asyncは関数宣言、awaitは「待つ」命令。
const data = await fetch('/api/users')
fetch
ブラウザからURLにHTTPリクエストを送る組み込み関数。APIからデータを取得するのに使う。
const res = await fetch('/api/data')
ビルド・デプロイ
ビルド
開発用のコードを本番環境向けに変換・最適化する処理。npm run build で実行する。
src/ → dist/ に変換される
バンドル
複数のJSファイルをブラウザが読みやすい1〜数個にまとめる処理。ビルドの一部。
100個のJSファイル → main.js 1個にまとめる
デプロイ
作ったアプリを本番サーバーに公開する作業。ローカルで動くものをインターネットに公開する。
vercel --prod でVercelに本番デプロイ
Vercel
フロントエンドアプリのホスティングサービス。GitHubと連携して自動デプロイできる。
vercel --prod でデプロイ / 無料プランあり
Vite
高速な開発ビルドツール。「npm run dev」で即座に開発サーバーが起動する。
npm create vite@latest my-app
Next.js
ReactベースのフルスタックWebフレームワーク。フロントエンドとバックエンドを1つで作れる。
npx create-next-app@latest my-app
フルスタック
フロントエンドとバックエンド両方を扱う開発スタイル。または両方含むフレームワークのこと。
Next.js + Supabase = フルスタック構成
CDN
世界中のサーバーにコンテンツを配置して高速化する仕組み。Vercelもこれを使っている。
日本ユーザーには日本のサーバーから配信される
Git / GitHub
Git
コードの変更履歴を管理するツール。いつ・誰が・何を変えたかを記録し、過去に戻れる。
git init でプロジェクトをGit管理下に置く
GitHub
Gitの履歴データをクラウドに保存・共有するサービス。チーム開発やバックアップに使う。
github.com/username/project でコードを公開管理
リポジトリ（repo）
Gitで管理するプロジェクトの保存場所。ローカル（自分のPC）とリモート（GitHub）がある。
gh repo create my-app でGitHubにリポジトリ作成
コミット
変更をGitに記録する操作。スナップショットを撮るイメージ。メッセージで内容を説明する。
git commit -m "ログインボタンを追加"
ステージング（git add）
コミットする変更を「選ぶ」操作。変更してもaddしないとコミットに含まれない。
git add app.js → このファイルをコミット対象に
プッシュ（push）
ローカルのコミット履歴をGitHub（リモート）に送信して同期する。
git push → GitHubに反映される
プル（pull）
GitHubの最新状態をローカルに取り込む。チームの変更を自分のPCに反映するときに使う。
git pull → 他の人の変更を取り込む
ブランチ
独立した作業ラインを作る機能。本番のコードを壊さずに新機能を開発できる。
git checkout -b feature/login で新ブランチ作成
マージ
あるブランチの変更を別のブランチに統合する操作。開発完了後にmainに取り込む。
git merge feature/login → mainブランチに統合
.gitignore
Gitから除外するファイルを指定するファイル。.envやnode_modules/は必ず書く。
.env / node_modules/ / dist/ を除外するのが基本
認証・セキュリティ
認証（Authentication）
「あなたは誰か」を確認するプロセス。ログイン・パスワード照合などがこれにあたる。
メールアドレス+パスワードでログイン = 認証
認可（Authorization）
「あなたは何ができるか」を管理するプロセス。管理者か一般ユーザーかの区別など。
管理者だけが削除ボタンを押せる = 認可
JWT（トークン）
ログイン情報を安全に伝えるための署名付き文字列。ヘッダー・ペイロード・署名の3部構成。
Authorization: Bearer eyJhbGci... ← APIに添付して送る
Cookie
ブラウザに保存される小さなデータ。サーバーが「このCookieを持ってきたらログイン済み」と判断する。
DevTools → Application → Cookies で中身を確認できる
セッション
ログイン状態を一定期間保持する仕組み。セッションが切れると再ログインが必要になる。
30分操作がないとセッション切れでログアウト
LocalStorage
ブラウザ内にデータを永続的に保存する場所。ページを閉じても消えない。JWTの保存先の一つ。
DevTools → Application → Local Storage で確認
脆弱性
攻撃者に悪用されうるセキュリティ上の弱点。Claude生成コードにも含まれることがある。
SQLインジェクション・XSS・ハードコードされたパスワード
SQLインジェクション
ユーザー入力を使ってDBへの不正な命令を実行させる攻撃。プリペアドステートメントで防ぐ。
name=' OR '1'='1 → 全データが漏洩するパターン
XSS（クロスサイトスクリプティング）
悪意あるスクリプトをページに埋め込んで他ユーザーのブラウザで実行させる攻撃。
<script>steal()</script> をコメント欄に投稿する攻撃
OWASP Top 10
Webセキュリティの国際標準団体OWASPが公表する「最も危険な脆弱性トップ10」のリスト。
インジェクション・認証の欠陥・XSSなどが毎年ランクイン
DNS・インフラ
DNS
ドメイン名をIPアドレスに変換する仕組み。インターネットの「電話帳」。
example.com → 192.0.2.1 に変換する
ドメイン
ウェブサイトの「名前」。人間が読めるサーバーの住所。DNSでIPアドレスに変換される。
example.com / my-app.vercel.app
IPアドレス
コンピュータのネットワーク上の住所。数字の羅列。人間にはドメインの方が覚えやすい。
192.0.2.1 / 2001:db8::1（IPv6）
DNS伝播
DNS設定の変更が世界中のサーバーに広がるまでの時間。最大48時間かかる。
ドメイン設定後「まだ見えない」のは伝播待ちのことが多い
キャッシュ
一度取得したデータを一時保存して高速化する仕組み。「直したのに変わらない」の主な原因。
Ctrl+Shift+R で強制再読み込みするとブラウザキャッシュが消える
JavaScript の基礎用語
変数
値に名前をつけて保存するもの。const は変更不可、let は変更可。
const name = "太郎" / let count = 0
関数
処理のまとまりに名前をつけたもの。呼び出すと実行される。
function greet() { ... } / const greet = () => { ... }
配列
複数の値を順番に並べたもの。[ ] で囲む。.map()・.filter()・.find()などのメソッドで操作する。
const list = ["apple", "banana", "cherry"]
オブジェクト
名前（キー）と値のペアをまとめたもの。{ } で囲む。JSONはオブジェクトに似た形式。
const user = { name: "太郎", age: 20 }
メソッド
オブジェクトや配列が持つ関数。「.」でつないで呼び出す。
array.map() / string.split() / console.log()
undefined / null
「値がない」状態。undefinedは未定義、nullは意図的に空にしたもの。エラーの原因になりやすい。
Cannot read properties of undefined ← よく見るエラー
型（Type）
データの種類。string（文字列）・number（数値）・boolean（真偽値）・object・arrayなど。
typeof "hello" → "string" / typeof 42 → "number"
コールバック
別の関数に「引数として渡す関数」。処理が終わったら呼んでもらう仕組み。
array.map(item => item.name) の item => item.name がコールバック

Claude Code に頼む前に、この3問に答えるだけで選択肢が絞れる
Q1
ユーザーが作ったデータを保存・表示する機能が必要ですか？
（例：投稿・ログイン・マイページ）
Q2
JavaScriptで動きをつけたいですか？
（例：ボタンを押したら変化する・モーダルが開く）
Q2
サーバーはVercelなどのクラウドサービスを使いますか？
（自分でVPSやサーバーを立てるのではなく）
おすすめ
Next.js + Supabase
DB・認証・APIをすべてまとめて扱える定番構成。Claude Codeとの相性が最もよく、サンプルコードやドキュメントが豊富。初めてのフルスタック開発にはこれが最短ルート。
npx create-next-app@latest my-app
Claude Codeへの第一声：「Next.js App RouterとSupabaseを使って〇〇を作りたい。まずプロジェクト構成と必要なパッケージを教えて」
おすすめ
Next.js + Supabase（自前サーバー構成）
自前サーバーの場合でも Next.js をフロントに使い、Supabase をDBに使う構成が管理しやすい。ただし VPS の設定はやや複雑になる。
npx create-next-app@latest my-app
Claude Codeへの第一声：「Next.jsフロント・自前サーバーで〇〇を作りたい。DockerかNode.jsサーバーの構成どちらがいいか相談したい」
おすすめ
Vite + React
フロントだけでインタラクションがあるアプリ。バックエンドなしで動く。ポートフォリオ・計算ツール・ゲームなどに最適。シンプルで学習コストが低い。
npm create vite@latest my-app -- --template react
Claude Codeへの第一声：「Vite + Reactで〇〇を作りたい。まずコンポーネント構成を一緒に考えたい」

フロントエンドとバックエンド
「見える部分」と「見えない部分」の分業
CLAUDE CODE がこう言ったとき

核心

Webアプリはフロントエンド（ブラウザで動く・目に見える）とバックエンド（サーバーで動く・データを扱う）に分かれています。どちらの問題かを区別できると、Claudeへの指示が的確になります。

フロントエンド（見える側）
HTML・CSS・JavaScript
ブラウザの中で動く
ボタン・レイアウト・アニメーション

バックエンド（見えない側）
Python・Node.js・Goなど
サーバーのPC上で動く
データの保存・認証・計算
ソースコードは外から見えない
フロント 「ボタンの色がおかしい」「アニメーションが動かない」

プログラム同士が会話する「窓口」
CLAUDE CODE がこう言ったとき

核心

API（Application Programming Interface）とは、異なるプログラムが決まった形式でデータをやりとりする「窓口」です。飲食店のカウンターのように「注文（リクエスト）→料理が来る（レスポンス）」の仕組みです。

→

気象API

{ "city": "Tokyo", "temp": 24, "weather": "晴れ" }
HTTPメソッドの意味
GET /users/123 ← 取得（読む） POST /users ← 作成（新しく送る） PUT /users/123 ← 更新（全部書き換え） DELETE /users/123 ← 削除
APIキーとは？

APIを使う「合言葉」です。絶対に公開しない（GitHubに上げない）のが鉄則。Claude Codeが「.envファイルに書いて」と言うのはこのためです。

Claude Codeへの応用：「401 Unauthorized」はAPIキーの問題。「429 Too Many Requests」は使いすぎ。エラーコードを伝えるだけで解決策が変わります。

7
データベースの基礎
データを「表」に整理して保存する仕組み
CLAUDE CODE がこう言ったとき
posts テーブルに title, content, user_id, created_at カラムを作成しました。
どんなデータがどこに入るか追える。「user_idとusersテーブルは紐づいているの？」「created_atで並び替えたい」という指示が自然に出せる。
核心

データベース（DB）はデータを整理して保存する仕組みです。Excelのようなテーブル（表）に、行（レコード）と列（カラム）でデータを管理します。このプロジェクトで使っているSupabaseもデータベースです。

テーブルのイメージ（usersテーブル）
| id | email | name | created_at | |----|-------------------|--------|---------------------| | 1 | taro@example.com | 太郎 | 2026-01-15 10:00:00 | | 2 | hana@example.com | 花子 | 2026-01-16 12:30:00 | ↑列（カラム） ↑行（レコード）
SQL — DBへの命令文
-- データを取得 SELECT * FROM users WHERE id = 1 -- データを追加 INSERT INTO users (email, name) VALUES ('taro@example.com', '太郎') -- データを更新 UPDATE users SET name = '次郎' WHERE id = 1 -- データを削除 DELETE FROM users WHERE id = 1
Supabaseとの関係

SupabaseはPostgreSQLというデータベースをAPI経由で使えるようにしたサービスです。コードから supabase.from('users').select() と書くと、裏で SELECT * FROM users が実行されています。

Claude Codeへの応用：「データが取得できない」と言う場合、「usersテーブルのselectが空配列を返す」と言うほうが格段に的確です。テーブル名とカラム名を伝えましょう。

8
非同期処理（async/await）
「待ち時間」のあるプログラムの書き方
CLAUDE CODE がこう言ったとき
データが undefined になっています。
fetch の前に await が抜けていました。修正しました。
なぜundefinedになったか・なぜawaitが必要かがわかる。次に同じエラーが出たとき「awaitの問題？」と自分で気づいて伝えられる。
核心

APIやDBへの通信には時間がかかります。その間プログラムが止まらないように「後で結果が来たら続きをやる」という仕組みが非同期処理です。async/awaitはそれを読みやすく書く構文です。

なぜ非同期が必要か
// 同期（悪い例）— APIが返るまで画面が固まる const data = fetch('/api/users') // ← これは待てない console.log(data) // ← まだ来ていないのに使う → undefined // 非同期（正しい）— awaitで「来るまで待って」と指示 async function getUsers() { const response = await fetch('/api/users') ← 待つ const data = await response.json() ← また待つ console.log(data) ← データが来てから実行される }
よくある失敗パターン
// awaitを忘れた → dataがPromiseオブジェクトになる const data = fetch('/api/users') console.log(data) // → Promise { <pending> } と表示される // async関数の外でawaitを使った → SyntaxError const data = await fetch('/api') // ← async関数の中でしか使えない
Claude Codeへの応用：「データがundefinedになる」「Promise {''}と表示される」というエラーは、ほぼ100% awaitの書き忘れです。Claudeに「awaitの問題ですか？」と聞くだけで解決することが多いです。

「開発用コード」を「本番用コード」に変換する仕組み
CLAUDE CODE がこう言ったとき
npm run build が失敗しました。dist/ フォルダが生成されていません。
ビルドとは何か・失敗すると何が起きるかわかる。「ローカルでは動くのに本番で動かない」という報告に「ビルドエラーかも」と気づけるようになる。
核心

ビルドとは、開発用のコードを本番環境向けに変換する処理です。バンドルとは、100個のJSファイルをブラウザが読みやすい1〜数個にまとめること。npm run buildがこれをやっています。

ビルドで何が起きているか
【開発時のコード】読みやすさ優先 src/ components/Button.jsx ← JSXを使ったコンポーネント utils/format.js ← ヘルパー関数 pages/index.jsx ← メインページ npm run build ↓ (変換・圧縮・最適化) 【ビルド後】速度・サイズ優先 dist/ index.html ← 1ファイルに統合 assets/main.a3f9b2.js ← 圧縮・難読化済み assets/style.8c2d1e.css
開発環境（dev）
コードが読みやすい
エラーの場所がわかる
変更が即座に反映される
ファイルサイズは大きい
本番環境（build）
コードが圧縮・難読化される
高速・軽量
変更には再ビルドが必要
Vercelにはこちらをデプロイ
Claude Codeへの応用：「ローカルでは動くのに本番で動かない」はビルドの問題であることが多いです。その一言を添えるだけで調査方向が変わります。

GitHubを使ったバージョン管理
コードの変更履歴を管理し、過去にいつでも戻れる仕組み
CLAUDE CODE がこう言ったとき
コミットしました："feat: ログイン機能を実装"
何が保存されたか・この時点にいつでも戻れる安心感がある。「さっきの変更前に戻して」「新機能は別ブランチで作ろう」という判断が自然にできる。
核心

Gitはコードの変更履歴を記録する「タイムマシン」です。「いつ・誰が・何を変えたか」をすべて記録し、問題が起きたら過去の状態に戻せます。GitHubはその履歴データをクラウドに置いてチームで共有するサービスです。

Git の基本的な流れ
① 変更する（ファイルを編集） app.js を修正 → まだ Git は知らない ② ステージング（コミット対象を選ぶ） git add app.js ← 「この変更を記録対象にする」 ③ コミット（スナップショットを保存） git commit -m "ログインボタンのバグを修正" → ローカルに履歴が保存される ④ プッシュ（GitHubに同期） git push → GitHub上にも反映される
ブランチ — 「作業用の分岐」

ブランチを使うと、本番のコードを壊さずに新機能の開発や修正ができます。完成したらメインブランチに統合（マージ）します。

git branch feature/login ← 新しいブランチを作成 git checkout feature/login ← そのブランチに移動 （コードを修正・コミット） git checkout main ← メインに戻る git merge feature/login ← 変更を統合する
.gitignore — Gitに無視させるファイル

APIキーや秘密情報が入った .env ファイルはGitに含めてはいけません。.gitignore に書いておくと自動的に除外されます。

# .gitignore の例 .env ← APIキーなどの秘密情報 node_modules/ ← 巨大なパッケージフォルダ dist/ ← ビルド成果物（再生成できる）
Claude Codeへの応用：「この変更をコミットして」「mainブランチに戻して」「前のコミットの状態に戻して」とそのまま伝えるだけで Git 操作を代行してくれます。エラーが出たらコミットIDとメッセージをそのまま渡しましょう。

11
認証の仕組み（Cookie・JWT）
「ログインしたことを覚える」仕組み
CLAUDE CODE がこう言ったとき
JWT の検証に失敗しています。Authorization ヘッダーが送られていません。
認証フローのどの部分で詰まっているか特定できる。「ログインはできているのにAPIが401になる」という状況の原因がJWTの送り忘れだとわかる。
核心

HTTPは毎回リセットされる仕組みです。なので「このユーザーはログイン済み」を覚えるための仕組みが別途必要です。それがCookieやJWTです。ログイン関連のバグの大半はここから来ます。

認証（Authentication）
「あなたは誰ですか？」
ログイン・パスワード確認
本人確認のプロセス
認可（Authorization）
「あなたは何ができますか？」
管理者か一般ユーザーか
アクセス権限の管理
JWTによるログインの流れ
① ログイン POST /api/login { email, password } ↓ サーバーがパスワードを照合 ↓ OKなら JWT（トークン）を発行して返す ② JWT = 3つの部分を「.」でつないだ文字列 eyJhbGci....eyJ1c2VyaWQ....SflKxwRJSMeK... ↑ヘッダー ↑ペイロード(中身) ↑署名(改ざん防止) ③ 以降のリクエストにJWTを添付 GET /api/profile Authorization: Bearer eyJhbGci... ↓ サーバーが署名を検証してOKなら返す
Cookieとは

ブラウザに保存される小さなデータです。サーバーが「このCookieを持ってきたらログイン済みとして扱う」という使い方をします。JWTをCookieに保存するパターンが一般的です。

Claude Codeへの応用：「ログインしても画面が変わらない」→「JWTが正しく保存されているかLocalStorageを確認して」と一言添えると調査が早くなります。DevTools → Applicationタブで確認できます。

DNS・ドメインの仕組み
URLがどうやってサーバーにたどり着くか
CLAUDE CODE がこう言ったとき
デプロイは成功しました。ただしカスタムドメインがまだ表示されていません。
DNS伝播待ち（最大48時間）か設定ミスかを区別できる。「しばらく待つ」か「設定を見直す」かの判断が自分でできる。
核心

ドメイン（例: example.com）は人間が読めるサーバーの住所です。コンピュータはIPアドレス（数字）しか理解できないので、DNS（Domain Name System）が「example.com → 192.0.2.1」と変換します。電話帳のような仕組みです。

URLを打ってからサーバーに届くまで
ブラウザに「example.com」を入力 ↓ ① DNSに問い合わせ 「example.comのIPアドレスを教えて」 ↓ ② DNSが答える 「192.0.2.1 だよ」 ↓ ③ そのIPアドレスにリクエスト GET / HTTP/1.1 → 192.0.2.1 ↓ サーバーがHTMLを返す
DNS伝播とは（今日詰まった場所）

DNSの設定変更は世界中のDNSサーバーに徐々に広がります（数分〜最大48時間）。Vercelにデプロイしても「見られない」ことがあるのはこのためです。「DNSが伝播中」という状態です。

よく見るドメイン関連の設定
Aレコード ドメイン → IPアドレス に変換 CNAMEレコード ドメイン → 別ドメイン に変換（Vercelでよく使う） TTL キャッシュの有効期限（秒）。短いと早く伝播する
Claude Codeへの応用：「デプロイしたのにURLが変わらない」「カスタムドメインが反映されない」は設定ミスかDNS伝播待ちのどちらかです。その区別をClaudeに伝えましょう。

キャッシュ
「直したのに変わらない」の原因の大半
CLAUDE CODE がこう言ったとき
CSSを修正してデプロイしました。反映には少し時間がかかる場合があります。
「直したのに変わらない」がキャッシュの問題とすぐ気づける。Ctrl+Shift+Rを試す→それでも変わらなければClaudeに報告、という流れが自分でできる。
核心

キャッシュとは、一度取得したデータを一時保存して「次回は保存済みを使う」仕組みです。表示が速くなる反面、更新してもキャッシュが残って古い状態が表示されることがあります。これが「直したのに変わらない」の原因の大半です。

キャッシュの種類と場所
① ブラウザキャッシュ あなたのPCに保存 → Ctrl+Shift+R で強制再読み込みで解決 ② CDNキャッシュ（Vercel等） 世界中のサーバーに保存 → 再デプロイするかキャッシュをパージ ③ データベースのキャッシュ クエリ結果を保存 → TTLが切れるか手動でクリア ④ アプリ内キャッシュ（React等） 状態管理の変数 → ページ再読み込みかstateのリセット
「直したのに変わらない」の診断フロー
Ctrl+Shift+R（強制再読み込み）で直った → ブラウザキャッシュが原因 再デプロイしたら直った → CDNキャッシュが原因 シークレットモードで開いたら直った → 拡張機能かブラウザキャッシュが原因 何をやっても変わらない → コード自体が変更されていない（保存し忘れ等）
Claude Codeへの応用：「修正したのに画面が変わらない」と言うとき、まずCtrl+Shift+Rを試してから報告しましょう。それだけで「キャッシュの問題でした」という無駄なやりとりが消えます。

何が危険か・なぜ危険かがわかる。「プリペアドステートメントに直して」という指示が自分で出せる。Claudeの警告を読み飛ばさずに対処できる。
核心

Claude Codeが生成するコードは「動く」コードですが、「安全な」コードとは限りません。代表的な攻撃パターンを知ることで、生成コードのレビューと修正ができるようになります。

SQLインジェクション — 最頻出の脆弱性
// NG: ユーザー入力をSQLに直接埋め込む（危険） const query = `SELECT * FROM users WHERE name = '${req.query.name}'` // 攻撃者が name=' OR '1'='1 を入力 → 全データが漏洩 // OK: プリペアドステートメント（安全） db.run('SELECT * FROM users WHERE name = ?', [req.query.name])
XSS — ブラウザ上で悪意あるスクリプトを実行させる
// NG: ユーザー入力をHTMLとして描画（危険） <div dangerouslySetInnerHTML={{ __html: userInput }} /> // 攻撃者が <script>steal()</script> を入力 → 他ユーザーで実行 // OK: テキストとして描画（自動エスケープ） <div>{userInput}</div>
機密情報の管理

Claude Codeとの会話はクラウドに送信されます。APIキーやパスワードを直接貼らず、YOUR_API_KEYのようなプレースホルダーで代用してください。生成されたコードにパスワードがハードコードされていたら必ず環境変数に移してください。

// NG: コードにパスワードを直書き password: "P@ssw0rd_2024!" // OK: 環境変数から取得 password: process.env.DB_PASSWORD
生成コードのレビューチェックリスト
✓ SQLにユーザー入力が直接入っていないか ✓ dangerouslySetInnerHTMLにユーザー入力を渡していないか ✓ パスワード・APIキーがコードに直書きされていないか ✓ 他人のデータにアクセスできる抜け穴（IDOR）がないか ✓ ログインAPIにレート制限があるか ✓ npm audit で依存パッケージに脆弱性がないか
Claude Codeへの応用：「このコードにセキュリティの問題がないかOWASP Top 10の観点でレビューして」と一言添えるだけで、脆弱性の洗い出しを自動化できます。特に認証・フォーム・APIエンドポイントで活用してください。
